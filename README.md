# 🎧 Viber - The Semantic Matchmaking Engine

> A multi-vector semantic search engine designed to match users based on a holistic blend of their **Music Taste**, **Personality**, and **Interests**.

[![Architecture: Multi-Vector](https://img.shields.io/badge/Architecture-Multi--Vector-blue?style=flat-square)](#)

## 🚀 Overview

This system goes beyond basic exact-match filtering. By leveraging dense vector embeddings and **Qdrant's native Multi-Vector Search**, we calculate a dynamically weighted semantic resonance between users. 

This architecture ensures we capture contextual and cultural overlaps—such as matching users with compatible "musical vibes" even if they share zero exact artists—without suffering from database recall loss.

---

## 🧠 Architecture

### 1. Vector Generation
Each user profile is translated into two distinct dense vector spaces before indexing:

* 🎵 **`music_vibe`**: A master centroid vector computed from the user's top Spotify tracks and artists (using Item2Vec or metadata-enriched LLM embeddings).
* 🧠 **`personality_interests`**: A singular, unified dense vector generated by baking the user's complete psychometric profile, survey responses, hobbies, and lifestyle text into a single cohesive string before passing it to the embedding model.

### 2. Storage Strategy (Qdrant)
We utilize **Qdrant** because it natively supports multiple named vectors per document. This allows us to store both multi-dimensional aspects of a user profile in a single record without splitting data across namespaces or collections.

**Example Payload:**
```json
{
  "id": "user_123",
  "payload": { 
    "name": "Alex", 
    "age": 22 
  },
  "vectors": {
    "music_vibe": [0.1, 0.4, 0.8, ...],
    "personality_interests": [0.9, 0.2, 0.5, ...]
  }
}
